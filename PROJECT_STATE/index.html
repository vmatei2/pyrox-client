Project State¶

Not tracked in git. Working notes on current state of the analytics workflow.

Hardening: rate limiting + error sanitization (2026-06-13)¶

  • pyrox_api_service/ratelimit.py (new): per-client-IP edge limiter behind a pure-ASGI RateLimitMiddleware, keyed off Fly-Client-IP. Added once, on the composed app (app.py), which wraps both the REST routes and the mounted /mcp sub-app — adding it to the sub-app too double-counts MCP requests. The /mcp 307→/mcp/ redirect is handled by charging a slot only on the served (non-redirect) response (test() to gate, hit() on a non-3xx response), so each logical call counts once. Header-less requests (in-process MCP→REST calls, tests) are exempt. Default 60/minute, override via PYROX_RATE_LIMIT. Uses the limits library (added to the api extra) directly — slowapi's SlowAPIMiddleware was rejected: it's route-endpoint coupled (unreliable on the mounted MCP sub-app) and raises AttributeError on the request-exempt path. Tested in tests/test_ratelimit.py (incl. a mount+redirect single-count guard).
  • app.py _raise_http: the DatabaseConfigurationError → 500 branch now logs the real cause and returns a generic "internal server error" instead of leaking the DuckDB path. 400/404 messages (mistake-recovery) unchanged.

Still deferred (decided 2026-06-13)¶

  • /mcp auth — endpoint stays open for now (bearer-token plan shelved).
  • Fly scale-to-zero / VM right-sizing — machine stays always-on for now.
  • DNS-rebind protection is wired (PYROX_MCP_ALLOWED_HOSTS) but the env var is not set in production yet.

MCP server (built 2026-06-13)¶

Goal: let Claude (and later Codex) answer natural-language questions against the HYROX dataset by connecting to a remote MCP server. Design captured in CONTEXT.md and docs/adr/0001-mcp-over-http-reporting-service.md.

What exists¶

  • GET /api/distribution (pyrox_api_service/app.py): the only new backend endpoint. Returns histogram bins + summary stats for a cohort.
  • gender required; season defaults to the latest season for the cohort; division defaults to open; metric defaults to overall.
  • Cohorts with n < 30 (DISTRIBUTION_SMALL_SAMPLE_MIN_N) come back with small_sample: true, suppressed tail percentiles (p10/p90 → null), and a note. Tested in tests/test_api.py (8 cases).
  • pyrox_api_service/mcp_tools.py: seven intent-shaped tool functions (list_filters, find_athlete, get_distribution, get_rankings, get_race_report, get_deepdive, get_athlete_profile). They call the REST app in-process via a Starlette TestClient (no network socket) and return error dicts instead of raising. List tools are compact (capped + total). Tested in tests/test_mcp_tools.py.
  • pyrox_api_service/mcp_app.py: registers the tools with FastMCP (stateless_http=True) and mounts the streamable-HTTP app at /mcp. Composed app served via pyrox_api_service.mcp_app:app (Dockerfile updated).
  • Dependency mcp>=1.2.0 added to the api extra in pyproject.toml.

Verified¶

  • Full suite: 94 passed, 3 skipped (integration). Ruff clean.
  • End-to-end MCP smoke: initialize → tools/list (7 tools) → tools/call list_filters returns real cohort data, with the REST API still live on the same app.

Not done / deferred¶

  • Pagination for list tools is approximated by an adjustable limit (the /api/rankings endpoint has no offset). Add real offset only if needed.
  • No auth (v1 decision). PYROX_MCP_ALLOWED_HOSTS/_ORIGINS enable DNS-rebinding protection if an operator wants to lock to the Fly host.
  • Rate-limiting now implemented at both edges (see the hardening section above).
  • small_sample threshold (30) and default division (open) were chosen from reasoning, not from the data — worth tuning against actual cohort-size distributions in pyrox_duckdb.
  • Claude-side plotting from returned bins is by design (no server-rendered images); not separately tested here.
