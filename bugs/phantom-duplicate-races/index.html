Bug report: phantom duplicate races (cross-division & cross-event)¶

  • Status: Open — root cause located in the data pipeline; fix not yet implemented.
  • Severity: High. Corrupts ~1.5% of race_results rows (~14.7k phantom rows) and silently inflates athlete profiles, rankings, distributions, and any cohort built from the affected events.
  • Discovered via: find_athlete("James Ingham") returning impossible races.
  • Area: Upstream scraper (separate repo, see §0) → source parquet in S3 → warehouse build in this repo. Not an API/MCP logic bug — the service faithfully serves corrupt source rows.
  • Date filed: 2026-06-16.

0. Exact file locations (two separate repos)¶

The data flows through two distinct repositories on disk. The bug originates in the first; the second only reflects it.

A. Upstream scraper repo — produces the source parquet in S3 (this is where the root cause lives): - /Users/vladmatei/VS_code_files/hyrox_analysis/scraping_code/hyrox_web_scraping.py — core scraper: discovers events, parses results per division/gender, writes one parquet per event to S3. - /Users/vladmatei/VS_code_files/hyrox_analysis/scraping_code/scraping_helpers.py - /Users/vladmatei/VS_code_files/hyrox_analysis/scraping_code/generate_manifest.py - /Users/vladmatei/VS_code_files/hyrox_analysis/scraping_code/dispatch_downstream.py - /Users/vladmatei/VS_code_files/hyrox_analysis/scraping_code/pipeline_reporting.py - /Users/vladmatei/VS_code_files/hyrox_analysis/scraping_code/hyrox_loader.py

B. This repo (pyrox-client) — reads that S3 parquet and builds the DuckDB artifact served by the API/MCP: - /Users/vladmatei/PycharmProjects/pyrox-client/scripts/ingest_duckdb_from_s3.py — orchestrates the DuckDB build. - /Users/vladmatei/PycharmProjects/pyrox-client/scripts/sql_queries.py — the SQL transforms (create_race_results_query, rankings, split percentiles). This faithfully copies whatever the parquet contains — no dedup. - /Users/vladmatei/PycharmProjects/pyrox-client/pyrox_duckdb — the built artifact (~1.1 GB). - /Users/vladmatei/PycharmProjects/pyrox-client/pyrox_api_service/reporting_queries.py + /Users/vladmatei/PycharmProjects/pyrox-client/src/pyrox/reporting.py — API/MCP read paths (correct; not the bug).

⚠️ Do not confuse the feeder with /Users/vladmatei/PycharmProjects/pyrox-client/scripts/scrape_hyresult_for_dates.py. 1. Symptom (what the user saw)¶

James Ingham (singles, GBR, athlete_id = 7ffd7bcf1bba0ad3e184ca89249f21b7) has 21 race rows, but several are clearly impossible: he appears to have run the exact same time (1:07:18) to the second at four different venues on four different dates, across every division (open / pro / doubles / pro_doubles) — at events he never physically competed in.

(roxzone 0:00:00 is itself implausible for a real singles race and is a secondary tell that these events are corrupt.)

3. The two (three) failure modes¶

The corruption is not a single bug; the same underlying mishandling shows up as distinct shapes.

Mode A — cross-division fan-out (largest)¶

Within one event, a single result is replicated once per division present in that event, each copy stamped with a different (mostly wrong) division but otherwise identical (same splits, same times).

Evidence — warsaw event LR3MS4JI1532:

division rows distinct names
doubles 653 653
open 653 653
pro 653 653
pro_doubles 653 653

All four divisions carry the same 653-athlete roster with identical singles splits (8 runs + roxzone present in all). A real event has different rosters per division (e.g. melbourne LR3MS4JI1182: doubles=2525 rows, open=919 rows — genuinely different). So warsaw's "doubles"/"pro_doubles" copies are the singles field mislabeled, not real doubles results.

Signature: in an affected event, rows ÷ distinct_names == number_of_divisions exactly (ratio 4.00 or 2.00).

Mode B — cross-event contamination (the one the user cares about)¶

The same roster is replicated across several distinct events (different event_id, location, start_date), so an athlete who raced once appears at venues they never attended — with identical splits. This is what produces James's 1:07:18 at brisbane + warsaw + cardiff + lisboa.

Mode C — duplicate location slug¶

The same physical event is ingested under two location slugs, e.g. johannesburg and johannesburg-i (both start_date = 2025-03-01). Because location is part of the result_id hash (see §4), the two copies get different result_ids and survive as separate rows.

4. Root cause analysis¶

The warehouse transform is innocent (but offers no guard)¶

result_id is built in /Users/vladmatei/PycharmProjects/pyrox-client/scripts/sql_queries.py (create_race_results_query):

md5(
    coalesce(event_id,'') || '|' || coalesce(trim(name),'') || '|' ||
    coalesce(division,'') || '|' || coalesce(total_time_str,'') || '|' ||
    coalesce(season,'') || '|' || coalesce(location,'') || '|' || coalesce(year,'')
) AS result_id

The transform reads straight from S3 parquet:

FROM read_parquet('s3://hyrox-results/processed/parquet/season=*/location=*/year=*/*.parquet',
                  hive_partitioning=true, union_by_name=true, binary_as_string=true)

There is no DISTINCT/dedup step. So: - Whatever duplication exists in the source parquet is copied 1:1 into race_results. - Because division and location are part of the md5, the same underlying performance under a different division (Mode A) or location slug (Mode C) produces a distinct result_id and is therefore not even a hash collision — it is a fully materialised extra row. - union_by_name=true across all partition files means two source files for the same event (Mode C, or a re-scrape) are concatenated, not merged.

Conclusion: the duplicates originate in the source parquet in S3, not in the DuckDB build. The upstream writer is /Users/vladmatei/VS_code_files/hyrox_analysis/scraping_code/hyrox_web_scraping.py. Relevant pointers in that file (line numbers as of 2026-06-16, ~1411 lines total — re-grep if they have drifted):

  • EVENT_VALUE_RE (~line 455) — ^(?P<division>[A-Z]+)_(?P<event_id>[A-Z0-9]+)$. The source site identifies a result set by DIVISION_EVENTID; event_id is shared across the divisions of one physical event.
  • discover_events_for_season (~line 542) — returns one row per (division, event_id), i.e. each division is a separate scrape target.
  • The per-event scraper class (~line 812) and its division×gender loop (for division, gender in all_div_gender, ~line 934) — fetches a roster for each division. This is the prime suspect for Mode A: if the source endpoint returns the complete event field regardless of the requested division (a known failure mode on result sites where the division filter is cosmetic), every division iteration captures the same finishers, each then stamped with the requested division. Verify by checking whether get_timings / the roster fetch actually constrains rows to the requested division.
  • Parquet writer (~lines 1080–1145) — file_name = f"{_slug(self.event_name)}__{self.event_id}.parquet", written under season={season}. The location partition the warehouse globs on (location=*) is derived downstream from event_name/event metadata. Mode C (johannesburg vs johannesburg-i) is two event_name/event_id files for the same physical event — confirm whether the source listed Johannesburg twice, or whether the manifest/dispatch step (generate_manifest.py / dispatch_downstream.py) emitted it twice.

Likely mechanisms by mode: - Mode A — division filter on the source returns the full field; the loop at ~line 934 captures it once per division. - Mode B — a batch of events shares one roster / the same roster associated with several event_ids (inspect discover_events_for_season output and generate_manifest.py). - Mode C — an event scraped/written twice under two event_name slugs.

Why this matters downstream¶

Every consumer treats race_results rows as independent finishers: - Rankings / percentiles (race_rankings, split_percentiles in /Users/vladmatei/PycharmProjects/pyrox-client/scripts/sql_queries.py) partition by division, gender, age_group — phantom rows inflate cohort sizes and distort event_size, percent_rank, and placement. - Athlete profiles count phantom races as real PBs/averages. - Distributions (/api/distribution) include duplicated finishers.

5. Scope / blast radius¶

Measured on the current pyrox_duckdb artifact (971,094 total rows):

  • ~14,707 phantom rows across 8,946 duplicate groups (grouping by event_id, name, total_time).
  • ~3,351 distinct athletes have at least one performance replicated across multiple locations.
  • 137 events contain at least some duplication; the corruption concentrates in a handful of wholesale-corrupt events.

Red flag — the "exactly 520" events (phoenix, washington-dc, emea-london-olympia, mechelen) each have precisely 520 distinct athletes × N divisions. A real Hyrox event roster is never a round 520 repeated across four cities. These specific event partitions look synthetic / seeded / capped-pagination in addition to being fan-out-duplicated, and should be inspected at the source before trusting any row from them.

6. Detection queries (reusable)¶

Mode A — events where every athlete is fanned across divisions:

SELECT event_id, any_value(location) AS location,
       COUNT(*) AS n_rows, COUNT(DISTINCT name) AS n_names,
       ROUND(COUNT(*)*1.0/NULLIF(COUNT(DISTINCT name),0),2) AS ratio,
       COUNT(DISTINCT division) AS n_divisions
FROM race_results
GROUP BY event_id
HAVING COUNT(*) > 200 AND ratio >= 1.9      -- ratio ~= n_divisions ⇒ corrupt
ORDER BY ratio DESC;

Mode B — same athlete + identical full split signature at >1 location (the authoritative per-row test):

WITH sig AS (
  SELECT name, total_time,
         run1_time||run2_time||run3_time||run4_time||run5_time||run6_time||run7_time||run8_time
         ||skiErg_time||sledPush_time||sledPull_time||burpeeBroadJump_time||rowErg_time
         ||farmersCarry_time||sandbagLunges_time||wallBalls_time AS splitsig,
         COUNT(DISTINCT location) AS n_locations,
         array_to_string(list_sort(array_agg(DISTINCT location)),' | ') AS cluster
  FROM race_results
  WHERE total_time_min IS NOT NULL
  GROUP BY name, total_time, splitsig
)
SELECT cluster, COUNT(*) AS n_signatures
FROM sig WHERE n_locations > 1
GROUP BY cluster ORDER BY n_signatures DESC;

Mode C — duplicate location slugs sharing a date:

SELECT start_date, COUNT(DISTINCT location) AS slugs,
       array_agg(DISTINCT location) AS location_slugs
FROM race_results
GROUP BY start_date
HAVING COUNT(DISTINCT location) > 1 AND
       bool_or(location LIKE '%-i') ;   -- '-i' / numeric suffixes are the tell

7. Recommended fix direction¶

Fixing only the DuckDB artifact treats the symptom; the source parquet must be corrected too, or the next rebuild reintroduces everything.

  1. Confirm at the source first. Inspect the S3 parquet for one affected event (e.g. warsaw event_id = LR3MS4JI1532). Determine whether the duplication is (a) one file with division-fanned rows, (b) multiple files per event, or (c) division-filter-ignored scrape responses. This decides whether the fix lives in the scraper or the manifest/dispatch step. Start in the scraper repo: /Users/vladmatei/VS_code_files/hyrox_analysis/scraping_code/hyrox_web_scraping.py (the division×gender loop at ~line 934 and the roster fetch it calls), then generate_manifest.py and dispatch_downstream.py in the same directory.

  2. Pitfalls — naive fixes that will not work:

  3. Dedup on result_id alone: does nothing. Every phantom already has a distinct result_id (division and location are in the hash). Modes A/B/C all survive a result_id dedup.
  4. "Keep one row per division": you cannot recover the true division — all copies are equally (in)valid. For Mode A the real division is unknown from the warehouse alone; it must come from the source.

  5. Dropping whole venues: over-deletes. James's cape-town row is genuine even though cape-town is in a contaminated cluster.

  6. A defensible warehouse-level guard (interim): dedup on the full content signature — (name, total_time, every split string, gender, age_group) — collapsing Mode B/C copies, and for Mode A reduce the per-division fan-out by detecting rows/names == n_divisions events. But this is lossy (it cannot reconstruct true divisions) and should be a stopgap, clearly logged, not the primary fix.

  7. Add a regression check to the ingest pipeline (/Users/vladmatei/PycharmProjects/pyrox-client/scripts/ingest_duckdb_from_s3.py) using the §6 queries: fail the build if any event has rows/names >= 1.9, or if any (name + splitsig) spans >1 location above a small threshold.

8. Open questions for the next agent¶

  • For each Mode B cluster, which event is the real one? (Is James's 1:07:18 a real brisbane/warsaw/cardiff/lisboa race that got copied, or fully fabricated? The warehouse cannot tell — needs the source/hyresult page.)
  • Are the "exactly 520" events real events at all, or seeded/sample data that leaked into the production artifact?
  • Why do the affected events all carry roxzone_time = 0:00:00? Is that a property of the corrupt source batch (useful as an extra detector), or coincidental?
  • In Mode C, the johannesburg dup pair differs only in nationality (None vs RSA, RSA) — confirming two source files, one enrichment pass apart. How many other events have this two-file shape?
