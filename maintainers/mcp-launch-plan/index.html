Pyrox MCP Launch Plan¶

Launch Position¶

Pyrox MCP is a public, read-only analytics connector for HYROX Race Results. The launch promise is narrow and defensible: Claude can answer common Result, Cohort, Distribution, Percentile, ranking, and athlete-profile questions through curated server-computed tools. It is not a raw SQL connector and not a replacement for the Python client.

The core architectural choice is already sound: MCP wraps the hosted FastAPI reporting Module over HTTP, exposing intent-shaped tools rather than the local DuckDB ReportingClient. That keeps Cohort definitions, Distribution calculations, Percentiles, thin-Cohort warnings, and row caps in one Implementation.

Launch Gate¶

Ship publicly when these checks pass:

  • The DuckDB artifact is rebuilt from guarded source data and the ingest integrity gate passes without ALLOW_DIRTY_INGEST=1.
  • pytest -q is green.
  • The deployed app serves pyrox_api_service.mcp_app:app, not the REST-only app.
  • /api/health returns healthy table and database metadata.
  • A live MCP smoke test can initialize the server, list tools, and call list_filters.
  • README contains a copy-paste connector command using https://pyrox-api.fly.dev/mcp/.
  • Maintainer docs state the open/no-auth decision, rate limit, DNS-rebinding host allow-list, and rollback path.

Phase 1: Harden the Public Connector¶

  1. Keep the public MCP Interface small. The seven current tools are enough for launch: list_filters, find_athlete, get_distribution, get_rankings, get_race_report, get_deepdive, and get_athlete_profile.

  2. Keep raw data behind curated tools. Do not expose SQL or arbitrary row export. This preserves locality for Cohort semantics and gives users consistent answers across REST, UI, and MCP.

  3. Rate-limit the open endpoint. Rate limiting is important because the MCP endpoint is public, open, and model-controlled. It is mainly a cost and availability guard, not a data-security mechanism. A simple launch default is enough: one shared per-IP window across REST and MCP, defaulting to 60/minute, configurable with PYROX_RATE_LIMIT.

  4. Enable MCP transport host protection in deploy config. Set PYROX_MCP_ALLOWED_HOSTS=pyrox-api.fly.dev for the public Fly app. Keep authentication out of v1 unless the project adds private data, write tools, or per-user quota/accounting.

Phase 2: Add the Try Pyrox MCP Page¶

Add a public docs page aimed at first-time users:

  • What Pyrox MCP can answer.
  • How to install it in Claude Code.
  • Example prompts:
  • "What seasons, divisions, and genders are available?"
  • "Find results for James Ingham."
  • "Show the distribution of female open finish times in season 8."
  • "Where would a 62-minute male open time rank in season 8?"
  • "Give me a race report for this Result and explain strongest/weakest Segments."
  • Expected caveats:
  • Results depend on public HYROX source data.
  • Thin Cohorts are flagged and should be treated cautiously.
  • Divisions are not pooled by default.
  • Claude renders charts from returned structured data.

Phase 3: Add a Live MCP Smoke Test¶

scripts/smoke_mcp.py accepts PYROX_MCP_URL and verifies the deployed connector:

  • Initialize MCP over streamable HTTP.
  • List tools and assert the expected seven tool names.
  • Call list_filters and assert seasons/divisions are non-empty.
  • Optionally call one stable aggregate query, such as a season 8 open Distribution.

Run it manually before launch:

PYROX_MCP_URL=https://pyrox-api.fly.dev/mcp/ uv run python scripts/smoke_mcp.py --json

The GitHub integration workflow runs the same smoke test against the live endpoint on main/manual runs, and the deploy workflow runs it immediately after Fly deploy. The script fails loudly but avoids large calls.

Deepening Opportunities¶

  1. MCP smoke Module Files: scripts/smoke_mcp.py, docs/maintainers/reporting-service.md, CI config. Problem: launch confidence currently depends on manual MCP client setup. The Interface is human memory plus README commands. Solution: create one executable smoke Module that hides MCP handshake details. Benefits: better locality for connector verification, and leverage for local, deploy, and scheduled checks through the same Interface.

  2. Public MCP docs Module Files: docs/mcp.md, mkdocs.yml, README.md. Problem: README has install commands but not a guided user journey or example prompts. Users must infer the Result discovery flow. Solution: make a dedicated Try Pyrox MCP page that teaches the search -> Result -> report flow and explains Cohort caveats. Benefits: more leverage from the existing tool docstrings, better locality for public-facing claims, and easier review before launch.

  3. MCP response contract Module Files: pyrox_api_service/mcp_tools.py, pyrox_api_service/reporting_queries.py, tests/test_mcp_tools.py. Problem: tool input schemas are good, but output schemas are absent because tools return bare dictionaries. That makes the Interface less explicit for MCP clients. Solution: after launch, introduce typed response contracts for the highest-value tools first: filters, distribution, rankings, and athlete search. Benefits: deeper tool Interfaces, better generated MCP metadata, and tests that pin public response shape rather than selected keys.

  4. Cohort explanation Module Files: pyrox_api_service/reporting_queries.py, docs. Problem: Cohort defaults are correct but spread across code, docstrings, and context notes. A caller must know several facts to interpret results safely. Solution: standardize every analytical response to include a compact Cohort explanation: filters applied, defaults used, n, and small_sample. Benefits: locality for analysis caveats and leverage across REST, UI, and MCP.

Launch Rollback¶

If the MCP endpoint behaves badly after launch, keep REST online and remove or block only the MCP mount in deploy config or by reverting the Docker entrypoint to pyrox_api_service.app:app. If traffic volume is the issue, lower PYROX_RATE_LIMIT first before disabling the connector.
